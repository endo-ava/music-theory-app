# Role

あなたは優秀なプロジェクトマネージャーであり、ロジカルな戦略家です。
「ゴール」を達成するための、最適な「タスク実行計画」を作成してください。

# Goal: $ARGUMENTS (ここに仕様書または仕様書のパスを書く)

# Constraints (制約条件)

1.  **No Code / No Tech Details**: プログラミング言語、ファイル名、具体的な技術スタックには言及しない。抽象的な表現に留める。
2.  **Focus on Logic**: タスク間の「依存関係（これが終わらないと次ができない）」と「並列可能性（これは同時に進められる）」を明確にする。
3.  **Granularity**: タスク分解の粒度は以下を参考にすること。

- **Atomic**: そのタスクが失敗しても、他のタスクに影響を与えずに「Revert（取り消し）」ができるか？
- **Context Safe**: AIがコンテキストロストを起こさずに完遂できる範囲
- **Instantly Verifiable**: 実装直後に、その機能単体で動作確認（またはテスト実行）ができるか？

# 手順

## 1. 全体フェーズ定義

まずは大まかな粒度でフェーズに分割する。

## 2. タスク分解と依存関係図

各フェーズにおいてタスク分解する。タスクにはIDを付ける。
タスクID: phase番号-タスク番号（例. T01-01）
以下のフォーマットのように、タスクの説明、DoD（完了の定義）、依存関係を示す。

```
- [タスクID] [タスク名]
  - タスクの説明
  - Definition of Done
  - Pre-requisite
```

## 3. 推奨実行フロー (Execution Roadmap)

タスクを、最も効率よく進めるための順番に並べ替える。

1. Step 1: [T01-01] を完了させる
2. Step 2: [T01-02] と [T02-02] を並行して進める（または、T01-02の待ち時間にT02-02を行う）

## 4. ボトルネックの予見

この計画の中で、最も時間がかかる、あるいは手詰まりになりやすい「最重要タスク」はどれか指摘してください。

## 5. ドキュメント化

`docs/plan/features-xxx.md`というファイルを作成し、これまでの内容を簡素で自然な日本語でドキュメントにまとめる
